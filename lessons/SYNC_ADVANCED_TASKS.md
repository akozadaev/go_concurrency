# Продвинутые задачи по пакету `sync` (методичка)

Прежде чем перейти к задачам, зафиксирую методологическую позицию: в конкурентном программировании отсутствие падений или отсутствие сообщений от `go test -race` **не доказывает** потокобезопасность. Это лишь отсутствие обнаруженных на данный момент нарушений. Каждая задача ниже сформулирована так, чтобы требовать эмпирической проверки, а не только логического рассуждения. Где поведение примитивов `sync` зависит от планировщика, GC или профиля нагрузки, указано, какие данные нужны для подтверждения корректности или эффективности решения.

Ниже 10 учебных задач, выстроенных по нарастанию сложности: в основном примитивы пакета `sync`, плюс в отдельных пунктах **`context`**, каналы и согласованное завершение горутин. Каждая включает цель, требования, метод верификации и примечание о границах применимости.

---

### Задача 1. Базовый потокобезопасный счётчик (`sync.Mutex`)

- **Формулировка:** Реализовать тип `Counter` с методами `Inc()`, `Dec()`, `Value()`. Методы должны быть безопасны при вызове из тысяч горутин.
- **Требования:** Запрет data races, **взаимное исключение** при изменении и чтении счётчика (сериализация доступа через `Mutex`; не смешивать с примитивами пакета `sync/atomic`, если в ТЗ явно не сказано иное), отсутствие паник.
- **Верификация:**
  - `go test -race -count=20`
  - Стресс-тест: 1000 горутин × 10⁴ операций - при **явно заданной схеме** (например, поровну `Inc` и `Dec`, либо известное смещение) проверить, что итоговое значение совпадает с ожидаемым (для симметричных `Inc`/`Dec` - нуль).
- **Примечание (отредактировано):** Утверждение "работает" без `-race` и без многократного прогона остаётся гипотезой. Race detector не гарантирует обнаружение всех гонок, но серия прогонов повышает достоверность. Проверка "итог равен нулю" имеет смысл **только если** в ТЗ зафиксирован баланс операций (например, равное число инкрементов и декрементов); иначе ожидаемое значение нужно выводить из явной модели нагрузки, а не подразумевать.

### Задача 2. Кэш с преобладанием чтения (`sync.RWMutex`)

- **Формулировка:** Реализовать `Cache` с методами `Get(key) (v, ok)`, `Set(key, v)`, `Delete(key)`. Оптимизировать под профиль 90% read / 10% write.
- **Требования:** Корректное разделение блокировок, отсутствие паник при конкурентном доступе.
- **Верификация:**
  - Бенчмарк `go test -bench=. -benchmem` с варьированием `readRatio`.
  - Сравнение с `sync.Mutex` при одинаковой нагрузке. Фиксация win/loss в зависимости от профиля.
- **Примечание:** `RWMutex` может деградировать при высокой конкуренции на запись или на системах с большим числом ядер. Вывод о "превосходстве" требует эмпирических данных, а не ссылки на документацию.

### Задача 3. Синхронизация группы горутин (`sync.WaitGroup`)

- **Формулировка:** Реализовать паттерн fan-out/fan-in: запустить N воркеров, каждый обрабатывает часть данных, результаты собираются в один канал. Дождаться завершения всех.
- **Требования:** Корректный порядок `Add()`, `Done()`, `Wait()`. Обработка отмены через `context.Context`. Закрытие результирующего канала только после завершения всех воркеров.
- **Верификация:**
  - По возможности - явная точка "все воркеры завершились" (дополнительный `WaitGroup`/`chan` в тесте) или `go.uber.org/goleak`; при эвристике с `runtime.NumGoroutine()` - сравнивать **дрейф между повторными прогонами**, а не равенство абсолютному baseline (служебные горутины рантайма, версия Go).
  - `go test -race -timeout=10s`
- **Примечание:** У нулевого `WaitGroup` счётчик **0**; в этом случае `Wait()` **сразу возвращается** - отсутствие вызовов `Add` само по себе **не** означает deadlock. Типичные ошибки: **пропуск `Done()`** (или лишний `Add`) → зависание на `Wait()`; **`Add` с положительной дельтой после того, как ожидание уже началось**, без согласованной модели повторного использования группы → гонка или panic по документации `WaitGroup`; отрицательный счётчик → **panic**. Без тестов на отмену контекста решение считается незавершённым.

### Задача 4. Ленивая инициализация (`sync.Once`)

- **Формулировка:** Реализовать загрузчик конфигурации, который выполняет дорогостоящую операцию (чтение файла/HTTP) ровно один раз при первом обращении.
- **Требования:** Потокобезопасность, корректная обработка ошибок инициализации.
- **Верификация:**
  - 1000 параллельных вызовов → счётчик реальных инициализаций должен равняться 1.
  - Тест на повторный вызов после ошибки: зафиксировать поведение и сравнить с требованиями ТЗ.
- **Примечание (отредактировано):** Семантику `sync.Once` нужно разделять по случаям. Если функция внутри `Do` **паникует**, `Once` **не** считается успешно завершённым: при следующем `Do` тело **вызовется снова**. Если функция **завершается без паники**, в том числе с **ошибкой в возвращаемом значении**, с точки зрения `Once` выполнение уже "было" - **повторного** вызова для новой попытки **не будет**. Поэтому требование "повторять при ошибке чтения/HTTP" **нельзя** удовлетворить одним "голым" `Once` без явной обёртки (собственный флаг повторов, другой тип-обёртка и т.д.). В методичке и ТЗ следует явно зафиксировать желаемую семантику при ошибке и при панике.

### Задача 5. Ограниченный буфер через `sync.Cond`

- **Формулировка:** Реализовать `BoundedBuffer` с блокирующими `Put(item)` и `Get()`. Размер буфера фиксируется при создании.
- **Требования:** Использование `Cond` с корректным `Wait()`/`Signal()`/`Broadcast()`. Отсутствие busy-wait. Блокировка только при полной/пустой очереди.
- **Верификация:**
  - Тест на deadlock с таймаутом 5 сек.
  - Проверка цикла условия: `for !canProduce/Consume { c.Wait() }`.
  - `go test -race` при 100 продюсерах и 100 потребителях.
- **Примечание:** Документация `sync.Cond` требует вызывать `Wait` **в цикле**, перепроверяя условие под мьютексом: после пробуждения состояние могло изменить другая горутина, возможны нюансы сигнализации; полагаться на "один `if` после `Wait`" нельзя. Производительность `Cond` часто уступает идиоматическому решению на каналах; выбор нужно обосновывать.

### Задача 6. Пул объектов (`sync.Pool`)

- **Формулировка:** Создать пул `[]byte` для сериализации/десериализации JSON. Измерить влияние на аллокации и время выполнения.
- **Требования:** Корректная инициализация `New`, безопасность использования между горутинами, учёт того, что объекты могут быть собраны GC в любой момент.
- **Верификация:**
  - Бенчмарк `testing.B` с замером `B.AllocsPerOp()`.
  - Сравнение с ручным пулом на канале фиксированного размера.
  - Запуск при разных `GOMAXPROCS` и интенсивном GC (`runtime.GC()` в цикле).
- **Примечание:** `sync.Pool` **не гарантирует** сохранение объектов между сборками мусора. Утверждение "снижает давление на GC" требует измерения в условиях, имитирующих рабочую нагрузку, а не в синтетическом тесте без GC.

### Задача 7. `sync.Map` vs `map` + `sync.RWMutex`

- **Формулировка:** Реализовать два варианта concurrent-мапы с одинаковым API. Провести сравнительное тестирование.
- **Требования:** Корректность при конкурентных read/write/delete. Фиксация метрик: пропускная способность, латентность, использование памяти.
- **Верификация:**
  - Бенчмарки при соотношениях read/write: 100/0, 90/10, 50/50, 0/100.
  - Анализ через `pprof` (`go test -bench=. -cpuprofile=cpu.pprof -memprofile=mem.pprof`).
- **Примечание:** `sync.Map` эффективен преимущественно при write-once/read-many или непересекающихся наборах ключей. В остальных случаях он часто медленнее. Выбор "лучшего" варианта без эмпирических данных является спекулятивным.

### Задача 8. Безопасное завершение воркеров (`context` + `WaitGroup` + каналы)

- **Формулировка:** Запустить пул из N воркеров, каждый читает из входного канала и пишет в выходной. Реализовать graceful shutdown по сигналу отмены контекста.
- **Требования:** Отсутствие гонок при закрытии каналов, корректный `wg.Wait()`, освобождение ресурсов, таймаут ожидания завершения. **`sync.Mutex` не обязателен** в этой постановке; применяйте его только если в вашем решении есть разделяемое состояние, требующее защиты (например, единственный флаг "закрыто" вместе с другими полями).
- **Верификация:**
  - После отмены контекста и ожидания завершения воркеров (с таймаутом, например ≤ 3 с) убедиться, что **нет устойчивого роста** числа горутин при **повторных** прогонах теста; избегать требования `runtime.NumGoroutine() == baseline` в абсолютном виде - в рантайме есть служебные горутины, а число может отличаться по версии Go и окружению. Предпочтительны явная синхронизация завершения (канал, `WaitGroup` в тесте) или библиотека уровня `goleak`, если разрешены зависимости.
  - `go test -race -timeout=10s`
  - Логирование порядка вызовов `close`, `wg.Done`, `wg.Wait`.
- **Примечание (отредактировано):** Порядок закрытия каналов и вызова `Done()` критичен; чтение из закрытого канала должно завершаться без паник. Проверка только через `NumGoroutine` с фиксированной паузой - **эвристика**, а не формальное доказательство отсутствия утечек; критерии нужно формулировать как отсутствие **дрейфа** при повторениях или как достижимость явных точек "все воркеры вышли".

### Задача 9. Рефакторинг кода с взаимной блокировкой

- **Формулировка:** Дана реализация, где две горутины захватывают два `sync.Mutex` в противоположном порядке, что ведёт к deadlock. Выявить проблему, устранить, сохранив семантику.
- **Требования:** Явное документирование порядка блокировок или отказ от вложенных lock'ов. Сохранение потокобезопасности.
- **Верификация:**
  - Динамический тест с таймаутом на deadlock.
  - Стресс-тест при `GOMAXPROCS=1,2,4,8`.
- **Примечание (отредактировано):** С **Go 1.18** у `sync.Mutex` есть **`TryLock()`**, но при неудаче он **не** создаёт отношения синхронизации с потоком, удерживающим мьютекс; для устранения классического deadlock’а по-прежнему опираются на **единый глобальный порядок** захвата нескольких мьютексов, на **один** мьютекс на инвариант, на каналы или на перестройку API. Утверждение "deadlock устранён" усиливается **тестом с таймаутом и многократными прогонами**, а не статическим анализом общего назначения: **`go vet` не обнаруживает взаимные блокировки по мьютексам** и не должен фигурировать как доказательство отсутствия deadlock. При необходимости упоминайте специализированные инструменты/обзор кода отдельно от критерия приёмки.

---

### Задача 10. Стресс-тест и валидация потокобезопасности

- **Формулировка:** Дана черновая реализация concurrent-структуры (например, ring buffer или priority queue). Написать набор тестов, запускающих 10⁵ операций параллельно, выявить скрытые гонки, исправить, задокументировать инварианты.
- **Требования:** Покрытие `go test -race`, бенчмарк, чек-лист проверок (deadlock, утечки, паники, порядок).
- **Верификация:**
  - Автоматизированный прогон в CI при разных `GOOS`/`GOARCH` (если доступно).
  - Анализ отчёта race detector.
  - Проверка стабильности при изменении `GOMAXPROCS` и под нагрузкой памяти.
- **Примечание:** Отсутствие репортов race detector **не является доказательством** потокобезопасности. Требуется многократный прогон, проверка инвариантов и, по возможности, ревью критических секций.

---

### Общие рекомендации по верификации

1. **Инструменты:** `go test -race -count=N`, `go vet`, `pprof`, `runtime.GOMAXPROCS`, `testing.B`, `context.WithTimeout`.
2. **Методология:** не доверять одному успешному прогону; фиксировать инварианты до/после конкурентных операций; разделять проверку корректности и производительности.
3. **Границы применимости:** поведение зависит от планировщика, версии Go и ОС; переносимость усиливается повторными прогонами на целевых платформах.

## **Общая цель программы**

Программа демонстрирует использование **контекста (`context.Context`)** в Go для управления временем выполнения операции и корректной обработки **таймаутов** или **отмены задачи**.  
Функция `task` моделирует длительную операцию (занимающую 2 секунды), но может быть прервана, если контекст сигнализирует о завершении (например, из-за истечения таймаута или явного вызова `cancel()`).

---

## **Импортируемые пакеты**

```go
import (
	"context"
	"fmt"
	"time"
)
```

- **`context`** - предоставляет тип `Context` и функции для передачи сигналов отмены, таймаутов и значений между горутинами.
- **`fmt`** - используется для вывода сообщений в консоль.
- **`time`** - для работы с задержками (`Sleep`) и таймерами (`After`).

---

## **Функция `task(ctx context.Context) (string, error)`**

```go
func task(ctx context.Context) (string, error) {
	select {
	case <-time.After(2 * time.Second):
		return "успешное завершение", nil
	case <-ctx.Done():
		return "", ctx.Err()
	}
}
```
(В репозитории ветки перечислены в таком порядке; пока сработал только один канал, результат тот же. Если **одновременно** готовы несколько `case` в `select`, выбирается один из них **псевдослучайно** - в учебных примерах с разными таймингами это редко, но об этом стоит помнить.)

### Назначение
Моделирует **долгую операцию**, которая:
- либо завершается успешно через **2 секунды**,
- либо прерывается досрочно, если контекст сообщает о завершении (например, по таймауту или отмене).

### Механизм работы
- Используется **`select`** - ожидание нескольких каналов.
- **Кейс `time.After(2 * time.Second)`**: по истечении 2 с с момента входа в `select` возвращается успех (если к этому моменту контекст ещё не отменён).
- **Кейс `ctx.Done()`**: канал закрывается при **отмене** или **дедлайне** контекста → возвращаются `""` и `ctx.Err()` (`DeadlineExceeded`, `Canceled` и т.д.).

> **Важно**: выбирается **любой готовый** `case`; если готовы несколько - выбор **псевдослучайный** (справедливый). Пока срабатывает только один канал, порядок перечисления `case` на результат не влияет.

---

## **Функция `main()`**

```go
func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second) // таймаут
	//ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second) // Успех
	time.Sleep(500 * time.Millisecond)
	//cancel()
	defer cancel()
	result, err := task(ctx)
	if err != nil {
		if err == context.DeadlineExceeded {
			fmt.Println("Превышен лимит ожидания")
		} else {
			fmt.Printf("Ошибка: %v\n", err)
		}
		//return
	}
	fmt.Println(result)
}
```

### Шаг 1: Создание контекста с таймаутом

```go
ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second)
```

- Создаётся **контекст с таймаутом 1 секунда** (отсчёт от момента вызова `WithTimeout`).
- По истечении срока контекст автоматически **отменяется**, `ctx.Done()` закрывается, а `ctx.Err()` обычно даёт `context.DeadlineExceeded`.
- `cancel()` нужно вызвать **один раз** после использования контекста (часто через `defer cancel()`), чтобы освободить таймер до утечки ресурсов - повторные вызовы безопасны.

> Закомментированная строка с **3 секундами** и отключённым досрочным `cancel`: задача `task` (2 сек) успевает завершиться успешно.

### Шаг 2: Задержка и ручная отмена (в текущем файле)

```go
time.Sleep(500 * time.Millisecond)
//cancel()
defer cancel()
```

- Сейчас в [main.go](main.go) вызов **`cancel()` закомментирован**: срабатывает именно **таймаут 1 с**, а не ручная отмена.
- Если **раскомментировать** `cancel()` после `Sleep`, контекст отменится с причиной **`context.Canceled`**, а не `DeadlineExceeded`. Тогда ветка `err == context.DeadlineExceeded` **не выполнится** - сообщение пойдёт в `fmt.Printf("Ошибка: ...")`. Для учебного кода логичнее явно различать оба случая (`errors.Is` / отдельные `if`).

### Шаг 3: Вызов задачи и обработка результата

```go
result, err := task(ctx)
```

- `task` стартует примерно через **500 мс** после создания контекста; **автоотмена** наступает на **1000 мс** от создания, то есть через ещё ~**500 мс** ожидания внутри `task`.
- До истечения таймаута готов канал только от `time.After(2s)` не будет; раньше сработает **`ctx.Done()`** → возвращается `ctx.Err()` (в этом сценарии - `DeadlineExceeded`).

### Шаг 4: Обработка ошибки

```go
if err != nil {
	if err == context.DeadlineExceeded {
		fmt.Println("Превышен лимит ожидания")
	} else {
		fmt.Printf("Ошибка: %v\n", err)
	}
}
fmt.Println(result)
```

- При **`DeadlineExceeded`** печатается строка про лимит ожидания.
- **`fmt.Println(result)` выполняется всегда**; при ошибке это даст **пустую строку** (лишний перевод строки). Обычно успех печатают только при `err == nil`.

---

## **Ожидаемое поведение при текущих настройках ([main.go](main.go))**

- Таймаут контекста: **1 с** от `WithTimeout`.
- **`time.Sleep(500 * time.Millisecond)`**, затем вызов **`task(ctx)`** (ручной **`cancel()`** отключён комментарием).
- "Успех" в `task` возможен только после **2 с** с момента входа в `select`, но контекст отменится раньше → срабатывает **`ctx.Done()`**, ошибка - как правило **`context.DeadlineExceeded`**.

###  Вывод программы:
```
Превышен лимит ожидания

```
(Вторая строка - пустая из-за `fmt.Println(result)` при пустом `result`.)

> Если переключиться на таймаут **3 с** (закомментированная строка) и не вызывать досрочный `cancel()`, `task` успеет получить значение из `time.After(2s)` и напечатает **`успешное завершение`**.

## **Ключевые концепции Go, продемонстрированные в коде**

| Концепция | Описание |
|---------|--------|
| **`context.Context`** | Стандартный способ передачи сигналов отмены и таймаутов между функциями и горутинами. |
| **`context.WithTimeout`** | Создаёт контекст, который автоматически отменяется по истечении заданного времени. |
| **`ctx.Done()`** | Канал, который закрывается при отмене контекста - используется для ожидания завершения. |
| **`ctx.Err()`** | Возвращает причину отмены (`context.Canceled` или `context.DeadlineExceeded`). |
| **`select` с каналами** | Позволяет конкурентно ждать нескольких событий (таймаут vs завершение операции). |
| **Ручная отмена через `cancel()`** | Позволяет досрочно прервать операцию по логике приложения. |

## **Потенциальные улучшения кода**

1. Оставить **`defer cancel()`** как основной способ завершения контекста; явный `cancel()` перед `defer` допустим, но для читаемости чаще либо только `defer`, либо `defer` + ранний выход с отменой по веткам логики.

2. Различать отмену и дедлайн через **`errors.Is(err, context.Canceled)`** / **`errors.Is(err, context.DeadlineExceeded)`** (вместо сравнения с `==`, если ошибка может быть обёрнута).

3. **Не выводить результат при ошибке**:
   ```go
   if err != nil {
       // ... обработка ошибки
   } else {
       fmt.Println(result)
   }
   ```

4. **Использовать `context.WithCancel`**, если не нужен таймаут, а только ручная отмена.

## **Общая цель программы**

Программа демонстрирует **управление жизненным циклом горутины** с помощью механизма **контекста (`context.Context`)**.  
Горутина `worker` выполняет периодическую работу (каждые 500 мс), но может быть **корректно остановлена** по сигналу отмены из основной функции `main`. Это - стандартный паттерн для безопасного завершения фоновых задач в Go.

---

## **Импортируемые пакеты**

```go
import (
	"context"
	"fmt"
	"time"
)
```

- **`context`** - предоставляет тип `Context` и функции для передачи сигналов отмены между горутинами.
- **`fmt`** - используется для вывода сообщений в консоль.
- **`time`** - для работы с задержками (`Sleep`) и таймерами (`After`).

---

## **Функция `worker(ctx context.Context)`**

```go
func worker(ctx context.Context) {
	for {
		select {
		case <-ctx.Done():
			fmt.Println("Получен сигнал отмены: ", ctx.Err())
			return
		case <-time.After(500 * time.Millisecond):
			fmt.Println("Работаю")
		}
	}
}
```

### Назначение
Это **фоновая горутина**, которая:
- бесконечно повторяет цикл,
- каждые **500 миллисекунд** выводит сообщение `"Работаю"`,
- но **немедленно завершается**, если получает сигнал отмены через контекст.

### Механизм работы

- **Бесконечный цикл `for {}`** - типичен для фоновых задач (например, обработчиков, слушателей, таймеров).
- **`select`** - ключевая конструкция для конкурентного ожидания нескольких каналов:
    - **Первый кейс**: `<-ctx.Done()`
        - Канал `ctx.Done()` **закрывается**, когда контекст отменяется (вызовом `cancel()`).
        - При получении сигнала:
            - выводится сообщение с причиной отмены (`ctx.Err()` - обычно `context.Canceled`),
            - функция **завершается** (`return`), и горутина останавливается.
    - **Второй кейс**: `<-time.After(500 * time.Millisecond)`
        - Создаётся **новый таймер** на каждую итерацию цикла.
        - Через 500 мс таймер срабатывает, и выводится `"Работаю"`.
        - Цикл повторяется.

> **Важно**: `time.After(500 * time.Millisecond)` создаёт **новый таймер на каждой итерации**. Это допустимо для простых случаев, но в производственном коде для периодических задач часто используют `time.Ticker`, чтобы избежать накопления неиспользуемых таймеров.

---

## **Функция `main()`**

```go
func main() {
	ctx, cancel := context.WithCancel(context.Background())
	go worker(ctx)
	time.Sleep(2 * time.Second)
	cancel()
	time.Sleep(1 * time.Second)
}
```

### Шаг 1: Создание отменяемого контекста

```go
ctx, cancel := context.WithCancel(context.Background())
```

- Создаётся **контекст, который можно отменить вручную**.
- `context.Background()` - корневой пустой контекст.
- Возвращается:
    - `ctx` - контекст, передаваемый в горутину,
    - `cancel` - функция, вызов которой **отменяет контекст** и сигнализирует об этом через `ctx.Done()`.

### Шаг 2: Запуск горутины

```go
go worker(ctx)
```

- Запускается горутина `worker`, которая начинает выполнять свою работу.
- Она получает контекст `ctx`, через который сможет узнать о необходимости завершиться.

### Шаг 3: Ожидание 2 секунды

```go
time.Sleep(2 * time.Second)
```

- Основная горутина (`main`) **приостанавливается на 2 секунды**.
- За это время `worker` успевает выполнить примерно **4 итерации** (2000 мс / 500 мс = 4 раза), выводя `"Работаю"`.

### Шаг 4: Отмена контекста

```go
cancel()
```

- Вызывается функция отмены - **сигнализирует горутине о необходимости завершиться**.
- Внутри `worker` канал `ctx.Done()` становится доступен для чтения, и горутина выходит из цикла.

### Шаг 5: Дополнительная задержка

```go
time.Sleep(1 * time.Second)
```

- Даёт **время горутине корректно завершиться** и вывести сообщение об отмене.
- Без этой задержки программа могла бы завершиться **сразу после `cancel()`**, и сообщение `"Получен сигнал отмены..."` **не успело бы напечататься** (так как `main` завершает всю программу).

>  Это **временное решение**. В реальных программах для ожидания завершения горутин используют `sync.WaitGroup` или каналы синхронизации.

## **Ключевые концепции Go, продемонстрированные в коде**

| Концепция | Описание |
|---------|--------|
| **Горутины (`go func()`)** | Легковесные потоки выполнения для параллельной работы. |
| **`context.Context`** | Стандартный механизм передачи сигналов отмены между горутинами. |
| **`context.WithCancel`** | Создаёт контекст, который можно отменить вручную. |
| **`ctx.Done()`** | Канал, закрывающийся при отмене контекста - используется для ожидания сигнала. |
| **`select`** | Позволяет ожидать несколько каналов конкурентно (работа vs отмена). |

## **Потенциальные улучшения**

1. **Использовать `time.Ticker` вместо `time.After` в цикле**:
   ```go
   ticker := time.NewTicker(500 * time.Millisecond)
   defer ticker.Stop()
   for {
       select {
       case <-ctx.Done():
           fmt.Println("Получен сигнал отмены:", ctx.Err())
           return
       case <-ticker.C:
           fmt.Println("Работаю")
       }
   }
   ```
   Это эффективнее, так как не создаёт новый таймер на каждой итерации.

2. **Синхронизация завершения через `sync.WaitGroup`** (вместо `time.Sleep`):
   ```go
   var wg sync.WaitGroup
   wg.Add(1)
   go func() {
       defer wg.Done()
       worker(ctx)
   }()
   // ...
   cancel()
   wg.Wait() // ждём завершения горутины
   ```

3. **Добавить `defer cancel()`** в `main` - хорошая практика на случай досрочного выхода (хотя в данном простом примере это не критично).

## **Общая цель программы**

Программа демонстрирует использование **контекста (`context.Context`)** для **передачи произвольных значений** (в данном случае - идентификатора) между функциями в Go.  
Это типичный паттерн для передачи метаданных (например, ID запроса, токенов аутентификации, трассировочных данных) **вглубь стека вызовов**, особенно в веб-серверах и распределённых системах.

---

## **Импортируемые пакеты**

```go
import (
	"context"
	"fmt"
)
```

- **`context`** - стандартный пакет Go для управления временем жизни операций, отмены и передачи значений.
- **`fmt`** - используется для форматированного вывода данных в консоль.

---

## **Функция `process(ctx context.Context)`**

```go
func process(ctx context.Context) {
	id := ctx.Value("ID")
	fmt.Println("process id: ", id)
}
```

### Назначение
Функция извлекает значение с ключом `"ID"` из переданного контекста и выводит его.

### Механизм работы
- **`ctx.Value(key interface{}) interface{}`** - метод контекста, возвращающий значение, связанное с заданным ключом.
    - Возвращает `interface{}`, поэтому результат **не типизирован**.
    - Если ключ не найден, возвращается `nil`.
- В данном случае ключ - строка `"ID"`, значение - строка `"121rdv-4df"`.
- Результат выводится через `fmt.Println`.

> **Важно**: `ctx.Value()` **не предназначен для передачи критически важных данных бизнес-логики**. Он используется **только для метаданных**, таких как:
> - ID трассировки (trace ID),
> - ID запроса (request ID),
> - данные аутентификации (user ID, токены),
> - локаль (язык, часовой пояс).

---

## **Функция `main()`**

```go
func main() {
	parentCtx := context.Background()
	ctx := context.WithValue(parentCtx, "ID", "121rdv-4df")
	process(ctx)
}
```

### Шаг 1: Создание родительского контекста

```go
parentCtx := context.Background()
```

- `context.Background()` - возвращает **пустой корневой контекст**, который **никогда не отменяется**.
- Используется как отправная точка для создания производных контекстов.

### Шаг 2: Добавление значения в контекст

```go
ctx := context.WithValue(parentCtx, "ID", "121rdv-4df")
```

- **`context.WithValue(parent, key, val)`** создаёт **новый контекст**, который:
    - наследует всё от `parentCtx`,
    - дополнительно хранит пару **ключ–значение**.
- **Ключ** - `"ID"` (тип `string`),
- **Значение** - `"121rdv-4df"` (уникальный идентификатор, например, ID запроса).

> **Рекомендация по ключам**:  
> Хотя в примере используется строка `"ID"` как ключ, **официальная документация Go рекомендует использовать типизированные ключи**, чтобы избежать коллизий между пакетами.  
> Пример правильного подхода:
> ```go
> type ctxKey string
> const requestIDKey ctxKey = "requestID"
> ctx := context.WithValue(parent, requestIDKey, "121rdv-4df")
> ```
> Это предотвращает случайное перезаписывание значений разными компонентами.

### Шаг 3: Вызов функции с контекстом

```go
process(ctx)
```

- Передаёт обогащённый контекст в функцию `process`.
- Функция извлекает значение и выводит его.

## **Ключевые концепции Go, продемонстрированные в коде**

| Концепция | Описание |
|---------|--------|
| **`context.Context`** | Унифицированный способ передачи данных, сигналов отмены и таймаутов через цепочку вызовов. |
| **`context.WithValue`** | Создаёт новый контекст с дополнительным значением по ключу. |
| **`ctx.Value(key)`** | Извлекает значение из контекста по ключу. |
| **Передача метаданных** | Контекст - стандартный способ "протаскивания" данных (например, ID запроса) без глобальных переменных или изменения сигнатур функций. |

---

## **Когда использовать `context.WithValue`?**

**Подходит для**:
- Логгирования (request ID, user ID),
- Трассировки (trace/span ID),
- Аутентификации (токены, роли),
- Локализации (язык, регион).

**Не подходит для**:
- Передачи обязательных аргументов функции,
- Хранения изменяемых данных,
- Передачи больших объёмов данных (контекст не для этого).

При соблюдении лучших практик (типизированные ключи, проверка `nil`, использование только для метаданных) этот паттерн становится мощным инструментом в разработке надёжных и масштабируемых приложений на Go.
